📋 Pattern MVC (Model-View-Controller)

💡 Points-clés avant de commencer : L'architecture qui révolutionne le web ▼

🎯 Objectif : Comprendre l'architecture MVC utilisée par 90% des frameworks web modernes pour structurer ses applications.

⚡ Points-clés à retenir :

  • Problème résolu : éviter le mélange présentation/logique/données dans le même fichier
  • Model = gère données et logique métier (BDD, calculs, règles)
  • View = interface utilisateur (HTML, CSS, templates)
  • Controller = chef d'orchestre qui coordonne Model et View

🔗 Dans votre métier :

Tous les frameworks PHP (Laravel, Symfony, CodeIgniter) et autres langages (Spring, Django, Rails) utilisent MVC. Le maîtriser vous ouvre toutes les technologies web modernes.

🚨 Exemple concret : Page e-commerce sans MVC = HTML + PHP + SQL dans le même fichier de 500 lignes. Avec MVC = 3 fichiers clairs de 50 lignes chacun. Lequel préférez-vous maintenir ?

Le problème à résoudre

Imaginez un site e-commerce : la présentation (HTML/CSS), la logique métier (calculs, règles) et les données (base de données) sont mélangées dans le même fichier. Résultat : un cauchemar à maintenir !

Data Access Object (DAO)

Points-clés avant de commencer : Abstraire l'accès aux données ▼

Objectif : Isoler l'accès aux données pour pouvoir changer de base de données sans impacter le code métier.

Points-clés à retenir :

  • Problème résolu : requêtes SQL éparpillées partout dans le code
  • Solution DAO : une classe par entité (UserDAO, ProductDAO) avec méthodes standards
  • Interface commune : save(), findById(), findAll(), delete()
  • Changement transparent : MySQL vers PostgreSQL sans toucher au code métier

Dans votre métier :

Le DAO permet de changer de base de données ou d'ORM sans tout casser. Essentiel pour les applications qui évoluent ou qui doivent supporter plusieurs environnements.

Exemple concret : Votre startup utilise SQLite en développement, MySQL en production, puis migre vers PostgreSQL. Avec DAO : changement d'une classe. Sans DAO : réécriture de 200 requêtes dans 50 fichiers.

Le problème

Votre code contient des requêtes SQL partout. Demain, vous passez de MySQL à PostgreSQL : tout est à réécrire !

Schéma DAO

LOGIQUE MÉTIER
(Controller/Model)
⬇️
DAO
Interface d'accès
⬇️
BASE DE DONNÉES
MySQL, PostgreSQL...

Principe du DAO

Abstraction : Le DAO cache les détails de la base de données

Interface unique : save(), findById(), delete(), findAll()

Changement transparent : On change juste l'implémentation du DAO

Structure conceptuelle du DAO

Interface standardisée : Le DAO définit un ensemble de méthodes standardisées pour manipuler les données : save(), findById(), findAll(), delete(), update(). Cette standardisation permet à n'importe quel développeur de comprendre immédiatement comment interagir avec les données.

Implémentations multiples : Derrière cette interface unique, vous pouvez avoir différentes implémentations : une pour MySQL, une pour PostgreSQL, une pour MongoDB, ou même une version de test qui stocke en mémoire. Le code métier reste inchangé.

Analogie du traducteur : Imaginez que vous dirigez une entreprise internationale. Plutôt que d'apprendre toutes les langues, vous employez des traducteurs spécialisés. Vous donnez toujours vos instructions en français (interface), et chaque traducteur (implémentation DAO) les adapte à sa langue cible. Vous pouvez changer de traducteur sans changer votre façon de communiquer.

Exemple de problème résolu : Une startup développe un MVP avec SQLite en local. En production, elle migre vers MySQL. Plus tard, pour des raisons de performance, certaines données migrent vers Redis. Sans DAO, chaque changement nécessite de réécrire toutes les requêtes dans tout le code. Avec DAO, seule l'implémentation de la couche d'accès change.

Cas d'usage typique en entreprise : Un système de gestion des ressources humaines doit fonctionner avec l'ancienne base Oracle du client et la nouvelle base PostgreSQL de l'entreprise qui rachète. Le DAO permet de créer deux implémentations sans modifier la logique métier des fiches employés, calculs de paie, etc.

Pattern Singleton

Points-clés avant de commencer : Une seule instance pour une seule ressource ▼

Objectif : Garantir qu'une classe n'a qu'une seule instance dans toute l'application et fournir un point d'accès global.

Points-clés à retenir :

  • Problème résolu : éviter de créer plusieurs connexions DB, configurations, ou logs
  • Principe : constructeur privé + méthode statique getInstance()
  • Cas d'usage : connexion DB, configuration, cache, système de logs
  • Attention : peut rendre le code difficile à tester (dépendance globale)

Dans votre métier :

Très utile pour les ressources coûteuses (connexions DB) ou les objets qui doivent être uniques (configuration de l'app). Mais à utiliser avec modération.

Exemple concret : Sans Singleton, votre application ouvre 50 connexions à la base de données. Avec Singleton : une seule connexion partagée. Économie de ressources importante !

Le problème

Vous ouvrez 50 connexions à la base de données dans la même page. Le serveur sature ! Il faut UNE SEULE connexion partagée.

Schéma Singleton

SINGLETON
Une seule instance
getInstance()

❌ new MaClasse() interdit

✅ MaClasse::getInstance()

Cas d'usage typiques

  • Connexion BDD : Une seule connexion pour toute l'application
  • Configuration : Un seul objet contenant les paramètres
  • Logger : Un seul système de logs
  • Cache : Une seule instance de cache en mémoire

Mécanisme du Singleton

Constructeur privé : La clé du pattern Singleton réside dans l'interdiction de créer des instances avec l'opérateur new habituel. Le constructeur est rendu privé, ce qui force l'utilisation d'une méthode statique spéciale (généralement getInstance()) pour obtenir l'unique instance.

Instance statique : La classe maintient une référence statique vers sa propre instance unique. La première fois qu'on demande l'instance, elle est créée. Les fois suivantes, la même instance est retournée.

Analogie du maire de ville : Dans une ville, il ne peut y avoir qu'un seul maire à la fois. Vous ne pouvez pas "créer" un nouveau maire à volonté. Si vous voulez parler au maire, vous passez par la mairie (getInstance()) qui vous met en contact avec l'unique maire en fonction.

Problème de conception résolu : Imaginez un système de cache global pour votre application. Sans Singleton, différentes parties du code pourraient créer plusieurs caches indépendants, perdant l'efficacité du partage des données en mémoire et créant des incohérences.

Cas d'usage controversé : Le Singleton est parfois considéré comme un "anti-pattern" car il introduit une dépendance globale difficile à tester. Dans un système de connexion base de données, il peut être préférable d'utiliser l'injection de dépendances plutôt qu'un Singleton pour faciliter les tests unitaires.

Exemple de projet problématique : Un développeur utilise un Singleton pour gérer la configuration de l'application. Le problème survient quand il faut tester différentes configurations : le Singleton rend difficile l'isolation des tests car ils partagent tous la même instance de configuration globale.

⚠️
Attention : Le Singleton peut rendre le code difficile à tester car il crée des dépendances globales. À utiliser avec parcimonie.

Classes et Objets

Points-clés avant de commencer : Le moule et l'objet créé ▼

Objectif : Comprendre la différence fondamentale entre une classe (le modèle) et un objet (l'instance) pour structurer efficacement son code.

Points-clés à retenir :

  • Classe = plan de construction, modèle, template (comme un plan d'architecte)
  • Objet = instance créée à partir de la classe (comme une maison construite selon le plan)
  • $this = référence à l'objet courant dans les méthodes
  • new = mot-clé pour créer une nouvelle instance

Dans votre métier :

Bien distinguer classe et objet évite les confusions fréquentes chez les débutants et permet de mieux concevoir ses applications.

Analogie concrète : Classe "Voiture" = les plans du constructeur. Objets = votre Clio rouge, la BMW de votre voisin, la Tesla du parking. Même modèle, instances différentes.

Classe vs Objet : L'analogie du moule

Classe : Le moule à gâteau (le plan, la définition)

Objet : Le gâteau créé avec le moule (l'instance)

Classe "Voiture" → Objets : ma Clio, ta BMW, sa Tesla

🏗️

Encapsulation et Visibilité

Points-clés avant de commencer : Protéger et contrôler l'accès aux données ▼

Objectif : Maîtriser l'encapsulation pour créer des objets robustes qui protègent leurs données et offrent un contrôle fin sur leur utilisation.

Points-clés à retenir :

  • Encapsulation = cacher les détails internes et contrôler l'accès aux données
  • Public = accessible partout (interfaces, méthodes principales)
  • Private = accessible uniquement dans la classe (données sensibles)
  • Protected = accessible dans la classe et ses enfants (héritage)

Dans votre métier :

L'encapsulation évite les erreurs de manipulation et rend le code plus sûr. Essentiel pour les applications robustes et la collaboration en équipe.

Analogie : Votre voiture - vous pouvez utiliser le volant et les pédales (public), mais vous ne devez pas toucher directement au moteur (private). Le tableau de bord avancé est réservé au mécanicien (protected).

Le principe de l'encapsulation

L'encapsulation consiste à cacher les détails internes d'un objet et à contrôler l'accès à ses données.

Analogie : Votre voiture

  • Public : Volant, pédales → Vous pouvez les utiliser
  • Private : Moteur interne → Vous ne devez pas y toucher directement
  • Protected : Tableau de bord avancé → Seul un mécanicien peut modifier
🔓

Héritage et Concepts Avancés

Points-clés avant de commencer : Réutiliser et étendre le code existant ▼

Objectif : Comprendre l'héritage pour créer des hiérarchies de classes et réutiliser efficacement le code existant.

Points-clés à retenir :

  • Héritage = une classe enfant hérite des propriétés et méthodes de sa classe parente
  • extends = mot-clé pour créer une relation d'héritage
  • parent:: = accéder aux méthodes de la classe parente
  • Redéfinition = override des méthodes parentes pour adapter le comportement

Dans votre métier :

L'héritage évite la duplication de code et permet de créer des architectures flexibles. Très utilisé dans les frameworks pour étendre les fonctionnalités de base.

Attention : L'héritage peut créer des dépendances fortes. Privilégier la composition quand l'héritage devient complexe.

L'héritage : réutiliser et spécialiser

Principe fondamental : L'héritage permet à une classe enfant d'acquérir automatiquement toutes les propriétés et méthodes de sa classe parente, tout en ayant la possibilité d'ajouter ses propres spécificités ou de modifier le comportement hérité.

Hiérarchie naturelle : Dans la nature comme en programmation, on observe des relations d'héritage. Un mammifère hérite des caractéristiques générales des vertébrés, mais ajoute ses spécificités (allaitement, poils). Un chien hérite des caractéristiques des mammifères mais ajoute ses comportements spécifiques.

Analogie professionnelle : Considérez la hiérarchie dans une entreprise. Un employé de base a certaines compétences et responsabilités. Un manager hérite de toutes ces compétences d'employé (il sait faire le travail de base) mais ajoute des compétences spécifiques (gestion d'équipe, planification). Un directeur hérite des compétences de manager et ajoute les siennes (stratégie, vision globale).

Redéfinition et spécialisation : Une classe enfant peut redéfinir le comportement d'une méthode héritée pour l'adapter à ses besoins spécifiques. C'est comme un chef cuisinier qui prend une recette de base et l'adapte selon son style et sa clientèle.

Exemple de projet de gestion documentaire : Vous avez une classe Document avec des propriétés générales (titre, auteur, dateCreation) et des méthodes communes (ouvrir, fermer, sauvegarder). Ensuite, DocumentPDF hérite de Document mais ajoute des méthodes spécifiques (compresser, protegerParMotDePasse). DocumentWord hérite aussi de Document mais redéfinit la méthode sauvegarder pour gérer les formats Word spécifiques.

Concepts avancés de la POO

Le Polymorphisme

Définition : Le polymorphisme permet à des objets de différentes classes de répondre à la même méthode, mais chacun à sa manière. C'est la capacité d'un même nom de méthode à produire des comportements différents selon l'objet qui l'exécute.

Analogie de l'orchestre : Quand le chef d'orchestre dit "jouez", chaque musicien exécute cette instruction différemment : le pianiste appuie sur des touches, le violoniste passe l'archet, le trompettiste souffle. Même instruction, comportements différents selon l'instrument.

Avantage en projet : Dans un système de notification, vous pouvez avoir une méthode envoyer() commune à EmailNotification, SMSNotification, PushNotification. Chaque classe implémente cette méthode selon son canal, mais le code qui utilise ces objets n'a pas besoin de connaître les détails.

Classes abstraites

Concept : Une classe abstraite est un modèle incomplet qui ne peut pas être instancié directement. Elle définit une structure commune et impose certaines méthodes à ses classes filles, tout en pouvant fournir des implémentations partielles.

Analogie du patron couture : Un patron de vêtement est abstrait : il définit la forme générale mais vous devez choisir le tissu, les couleurs, les finitions pour créer un vêtement concret. Le patron impose certaines contraintes (emplacement des coutures) mais laisse des choix d'implémentation.

Interfaces

Définition : Une interface définit un contrat : elle spécifie quelles méthodes une classe doit implémenter, sans fournir l'implémentation. C'est une garantie que toute classe qui "signe" ce contrat fournira certains services.

Analogie de la norme : Une prise électrique est une interface : elle définit un standard (forme, voltage) que tout appareil doit respecter pour se brancher. Peu importe comment l'appareil fonctionne à l'intérieur, s'il respecte l'interface, il peut se connecter au réseau.

Cas d'usage typique : Dans un système de paiement e-commerce, vous définissez une interface PaiementProcessor avec une méthode traiterPaiement(). Ensuite, PayPalProcessor, StripeProcessor, BankTransferProcessor implémentent cette interface, chacun à sa manière, mais le système de commande peut les utiliser de façon uniforme.
